# **MPAP — Multi‑Plane Attachment Protocol**  
*A protocol for creating, maintaining, and validating attachments across multiple planes without collapsing them.*

---

## **1. Purpose**

MPAP defines the **protocol** for attaching documentation, notes, diagrams, and conceptual materials to components across multiple planes while preserving:

- plane separation  
- non‑collapse  
- non‑interpretation  
- boundary integrity  
- traceability  

MPAP ensures that attachments **support** the architecture without becoming part of it.

---

## **2. What MPAP Is**

MPAP is:

- a **protocol**, not a plane  
- a **method of attachment**, not a structural layer  
- a **documentation rule‑set**, not a governance system  
- a **non‑interpretive scaffold**, not a meaning‑bearing entity  

MPAP exists to keep documentation **clean**, **separated**, and **non‑intrusive**.

---

## **3. What MPAP Prevents**

MPAP prevents:

- collapse between planes  
- interpretive drift  
- accidental authority  
- documentation becoming structure  
- structure becoming documentation  
- maps being mistaken for models  
- attachments being mistaken for planes  

---

## **4. Attachment Protocol**

Every MPAP attachment must follow these rules:

### **4.1 Declare Its Category**
Each attachment must declare itself as one of:

- **Structural Attachment**  
- **Governance Attachment**  
- **Ethical Attachment**  
- **Documentation Attachment**  
- **Meta‑Attachment** (maps, diagrams, conceptual overviews)

This prevents category drift.

---

### **4.2 Declare Its Plane of Reference**
Each attachment must specify which plane it *references*, not which plane it *belongs to*.

Example:

```
Plane of Reference: Structural Plane
Attachment Category: Documentation Attachment
```

This prevents collapse.

---

### **4.3 Declare Its Boundaries**
Each attachment must state:

- what it describes  
- what it does NOT describe  
- what it supports  
- what it does NOT replace  

This prevents interpretive overreach.

---

### **4.4 Declare Its Scope**
Each attachment must specify:

- scope of description  
- scope of relevance  
- scope of limitation  

This prevents attachments from becoming normative.

---

### **4.5 Declare Its Non‑Authority**
Every attachment must include:

```
This attachment does not interpret, diagnose, evaluate, or prescribe.
It is a documentation scaffold only.
```

This keeps MPAP aligned with UMA ethics.

---

## **5. Attachment Structure Template**

Every MPAP attachment uses this structure:

### **5.1 Structural Notes**
What the component *is*:

- shape  
- position  
- relationships  
- boundaries  

### **5.2 Governance Notes**
What the component *allows or restricts*:

- constraints  
- permissions  
- invariants  

### **5.3 Ethical Notes**
What the component *protects*:

- dignity  
- sovereignty  
- non‑coercion  
- non‑interpretation  

### **5.4 Documentation Notes**
What the attachment clarifies:

- purpose  
- context  
- usage  
- limitations  

### **5.5 Meta Notes (Optional)**
Maps, diagrams, or conceptual overviews.

---

## **6. Validation Protocol**

Before an attachment is accepted, it must pass:

- **Plane Separation Check**  
- **Boundary Integrity Check**  
- **Non‑Authority Check**  
- **Scope Check**  
- **Interpretation Check**  
- **Collapse Check**  

If it fails any check, it must be revised.

---

## **7. One‑Sentence Essence**

**MPAP ensures that attachments support the architecture without ever becoming part of it.**
