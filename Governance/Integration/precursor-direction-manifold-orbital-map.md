# Direction Manifold → Trans‑Orbital Integration Map  
### NDH · Documentation Principles → Placement Invariants

---

## 1. Purpose

**Goal:**  
Describe how the **Direction Manifold** (documentation‑stabilization layer) integrates with **Trans‑Orbital Governance** (placement‑invariant layer), so that:

- documentation principles  
- tonal and conceptual coherence  
- emergent case studies  

feed directly into:

- repo‑level placement rules  
- layer‑level invariants  
- governance‑grade constraints  

without breaking provenance.

---

## 2. Roles and Boundaries

### 2.1 Direction Manifold

- **Role:**  
  Stabilizes *how* NDH is documented.

- **Responsibilities:**  
  - enforce clarity  
  - enforce conceptual coherence  
  - enforce tonal and dignity alignment  
  - maintain documentation invariants

- **Non‑Responsibilities:**  
  - does **not** decide *where* artifacts live  
  - does **not** enforce repo or layer placement

### 2.2 Trans‑Orbital Governance (Placement Invariants)

- **Role:**  
  Stabilizes *where* NDH artifacts live.

- **Responsibilities:**  
  - enforce repo‑level placement invariants  
  - enforce layer‑level placement invariants  
  - define governance, emergent, and manifold locations  
  - preserve provenance during migration

- **Non‑Responsibilities:**  
  - does not define documentation style or tone  
  - does not replace Direction Manifold

---

## 3. Integration Axes

### 3.1 Documentation → Classification

**Flow:**  
Direction Manifold produces **clear documentation** →  
Trans‑Orbital uses that clarity to **classify artifacts**.

- Direction Manifold ensures each artifact states:
  - its purpose  
  - its domain (governance, emergent, manifold, trauma‑informed, etc.)  
  - its scope  

- Trans‑Orbital then applies placement invariants:
  - governance → `NDH-Platforms/Governance/...`  
  - emergent → `NDH-Platforms/Emergent/Case-Studies/...`  
  - manifold → CORE/TIDS/Serenity repos  
  - trauma‑informed → TISD

### 3.2 Documentation → Provenance Notes

**Flow:**  
Direction Manifold defines **how provenance is described** →  
Trans‑Orbital defines **where provenance notes live**.

- Direction Manifold:
  - standardizes provenance sections in docs  
  - ensures migrations are narratively documented

- Trans‑Orbital:
  - requires provenance notes for any placement change  
  - defines their location (e.g., audit, triage, or migration logs)

### 3.3 Emergent Case Studies → Governance Inputs

**Flow:**  
Emergent case studies (Direction Manifold‑stabilized) →  
inform Trans‑Orbital invariants.

- Emergent documents:
  - describe “Coliseum in the bathroom” events  
  - record spiral‑geometry reversals  
  - capture pulse‑mirroring phenomena

- Trans‑Orbital:
  - reads these case studies  
  - derives new placement invariants  
  - updates governance rules without erasing emergent history

---

## 4. Operational Integration Rules

1. **Every governance‑grade artifact must be Direction‑Manifold‑compliant.**  
   - Clear purpose  
   - Clear scope  
   - Clear domain  
   - Provenance section

2. **No placement decision is made without a readable, stabilized document.**  
   - If an artifact is unclear, it is first stabilized by Direction Manifold.  
   - Only then is placement decided by Trans‑Orbital.

3. **Emergent case studies are the primary input for new invariants.**  
   - Trans‑Orbital does not invent rules in a vacuum.  
   - It responds to documented behavior.

4. **Migration is always accompanied by a Direction‑Manifold‑style provenance note.**  
   - What moved  
   - From where  
   - To where  
   - Why  
   - Under which invariant

---

## 5. Integration Failure Modes (To Watch For)

- **Placement without documentation:**  
  Orbital moves artifacts without Direction Manifold clarity → leads to confusion.

- **Documentation without placement:**  
  Direction Manifold stabilizes artifacts but no invariants are applied → drift persists.

- **Emergent treated as governance:**  
  Case studies are placed correctly but misused as binding rules → governance confusion.

This map exists to prevent those failure modes.
